# Phase quick-260525-j03 Plan 01: Add latency_ms tracking to each LLM provider call — Summary

**One-liner:** Monotonic wall-clock timing (hrtime) added around each provider `prompt()` call, persisted as `latency_ms` integer column with backward-compatible DTO extension and zero-config migration.

## What Was Built

Added per-provider latency tracking from end to end:

1. **Migration** (`2026_05_25_000001_add_latency_ms_to_diagnostic_results_table.php`): Plain `integer` column `latency_ms` with `default(0)` so ALTER works against existing rows.

2. **DiagnosticResult model**: Added `'latency_ms'` to `$fillable` and `'latency_ms' => 'integer'` to `casts()`.

3. **ProviderResult DTO**: Added `public readonly int $latencyMs = 0` constructor property and optional `int $latencyMs = 0` param to `fromPrismResponse()` — all existing call sites in tests require zero changes.

4. **PrismStructuredCaller**: Wrapped `DiagnosticAgent::prompt()` in `hrtime(true)` measurement; converts nanoseconds to integer milliseconds via `(int) round(... / 1_000_000)` and passes the value as `latencyMs:` into `fromPrismResponse()`.

5. **DiagnosticResultRepository**: Added `'latency_ms' => $dto->latencyMs` to the `DiagnosticResult::create([...])` array.

## Tests Added

- `ProviderResultTest`: default latencyMs=0 when not passed; exact value preserved when passed.
- `PrismStructuredCallerTest`: persisted row has `latency_ms` as a non-negative integer.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Constructor property ordering adjusted for PHP default-value constraint**

- **Found during:** Task 2 GREEN phase
- **Issue:** Plan placed `latencyMs` after `tokensOutput` (before `requestId`/`promptVersion`), but PHP requires all parameters with defaults to come after all parameters without defaults. Adding `latencyMs = 0` in that position would require giving `requestId` and `promptVersion` empty-string defaults, which is semantically wrong.
- **Fix:** Placed `latencyMs` at the end of the constructor after `promptVersion`, with `= 0` default. Named-argument syntax in `new self(...)` means position is irrelevant to callers.
