Stack real (composer.json):
- php: ^8.3 (ambiente atual roda PHP 8.4.1)
- laravel/framework: ^13.0  (Laravel 13 — NÃO Laravel 12)
- laravel/ai: ^0.6          (SDK oficial Laravel AI — NÃO prism-php/prism)
- laravel/tinker: ^3.0
- dev: pestphp/pest ^4.6, laravel/pint ^1.27, mockery, collision, phpunit ^12.5

Banco: SQLite (DB_CONNECTION=sqlite no .env.example)

Provedores LLM configurados em config/ai.php (os 4 do escopo + outros do scaffold):
- anthropic  -> model text.default = claude-sonnet-4-6   -> env ANTHROPIC_API_KEY
- openai     -> model text.default = gpt-4o              -> env OPENAI_API_KEY
- gemini     -> model text.default = gemini-3.5-flash    -> env GEMINI_API_KEY
- deepseek   -> model text.default = deepseek-chat       -> env DEEPSEEK_API_KEY
(Apenas essas 4 chaves estão no .env.example e são usadas pelo DiagnosticService.)

Rotas:
- routes/api.php:  POST /api/diagnose  -> App\Http\Controllers\DiagnoseController (single-action, __invoke)
- routes/web.php:  GET  /          -> view welcome
                   GET  /results   -> ResultsController (dashboard de comparação por exercício e por LLM)
                   POST /results/label -> StoreLabelController (rótulos persistentes nos grupos de exercício)

Endpoint POST /api/diagnose:
  Request (validado por DiagnoseRequest):
    - code: string (obrigatório) — código TypeScript com erro
    - statement: string (obrigatório) — enunciado do exercício
  Comportamento (DiagnosticService::run):
    - Gera um request_id (UUID)
    - Dispara os 4 provedores EM PARALELO via Illuminate\Support\Facades\Concurrency
    - Cada falha de provedor é absorvida (Log::warning) e omitida do array (resultado parcial)
    - Cada resultado bem-sucedido é PERSISTIDO em diagnostic_results antes do retorno
  Response:
    - 200: array JSON (0 a 4 itens), um objeto por provedor que respondeu, com os campos:
        provider, model, diagnosis, pc3_category, feedback, confidence,
        tokens_input, tokens_output, request_id, prompt_version
      (Observação de precisão: o DTO ProviderResult também carrega error_code e latency_ms,
       que são persistidos no banco, mas o JSON de resposta do controller atual NÃO os inclui.
       Documente apenas os campos realmente retornados pelo endpoint.)
    - 503: {"message": "All providers failed"} quando todos os provedores falham

Taxonomia PC³ (enum App\DTOs\Pc3Category): Predicate, Concept, Context
error_code (enum App\DTOs\ErrorCode, persistido no banco): B6, B8, B9, B12, C1, C3, C8, G3, G4, H1, NONE

Persistência — tabela diagnostic_results (migrations):
  Colunas: id, provider, model, diagnosis (text), pc3_category (enum Predicate/Concept/Context),
           feedback (text), confidence (float), tokens_input (int), tokens_output (int),
           request_id (uuid, indexado), prompt_version, error_code (enum, default NONE),
           latency_ms (int), created_at, updated_at
  4 linhas por request_id (uma por provedor) — é o corpus de pesquisa reproduzível.

Scripts composer (composer.json):
  - composer setup : install + copia .env + key:generate + migrate --force + npm install + npm run build
  - composer dev   : sobe server + queue:listen + pail + vite (concurrently)
  - composer test  : config:clear + artisan test (Pest)

Escopo (PROJECT.md): MVP — sem autenticação, sem frontend além do dashboard /results, apenas o endpoint POST /api/diagnose.
